# Project Research — FEATURES dimension

**Domain:** 前端页面级 AI 验证平台
**Milestone:** Greenfield
**Date:** 2026-07-09

---

## 功能分类(按价值与复杂度)

### 一、输入与解析(两个入口场景)

#### Table stakes(必须有,否则产品不成立)
- **F-IN-01 文本粘贴输入**:粘贴问题单文本 / 需求验收用例文本进来
- **F-IN-02 文件上传**:支持上传 .md / .txt / .docx / .xlsx(用例常以这些格式存在)
- **F-IN-03 AI 解析输入**:LLM 把自由文本拆解为结构化字段——验证目标、涉及页面、涉及元素、期望结果、验收点
- **F-IN-04 输入分类路由**:自动判断这张单子是"回归断言型"(走脚本)还是"根因诊断型"(走 Agent 探索)

#### Differentiators(差异化)
- **F-IN-10 结构化录入兜底**:AI 解析不出来时,用户手填字段补全
- **F-IN-11 解析结果可编辑/可纠错**:AI 解析完给用户确认,可改

**Anti-features(明确不做)**
- ❌ 对接 Jira/Tapd/飞书等工单系统 API —— v1 通过粘贴/上传,用户已确认

---

### 二、术语库(命脉 —— AI 自主解析"该进哪个页面"的基础)

#### Table stakes
- **F-GL-01 术语条目 CRUD**:增删改查,每条 = {术语, 别名[], 目标URL, 入口路径, 关键元素定位器[]}
- **F-GL-02 术语检索**:AI 解析问题时,按问题单出现的词命中术语库,返回对应页面/定位器
- **F-GL-03 术语-页面绑定**:一个术语可关联到一个页面及其关键元素

#### Differentiators
- **F-GL-10 术语自动学习**:用户手动把问题单里的词"存为术语"时,引导补全 URL/定位器(可借助录制)
- **F-GL-11 别名/同义词管理**:同一页面多种叫法(如"工单列表"/"ticket list"/"ticket-list-page")
- **F-GL-12 术语命中可视化**:解析时高亮问题单里命中的术语,让用户看到 AI 用了哪些映射

**Anti-features**
- ❌ 术语库自动从代码/页面 DOM 反推 —— 太脆且不准,人工维护更可靠

---

### 三、用例库(可复用、可一键复跑 —— "别人二次验证"的载体)

#### Table stakes
- **F-CASE-01 用例存储**:每条用例 = 一份"验证规约"(导航+前置+操作+断言集+证据规范)
- **F-CASE-02 用例检索**:按页面/术语/关键词/标签检索已有用例
- **F-CASE-03 用例一键执行**:选中用例 → 在本地 Chrome 跑 → 出 pass/fail
- **F-CASE-04 用例与问题单/需求关联**:可追溯到来源

#### Differentiators
- **F-CASE-10 用例版本化**:页面改版导致定位器失效时,保留历史版本
- **F-CASE-11 用例导出**:导出为独立 Playwright 脚本文件(可脱离本应用运行,真·交给别人)
- **F-CASE-12 用例标签/分组**:按功能模块、按环境、按优先级

**Anti-features**
- ❌ 用例云同步/团队共享 —— 单用户本地,v2 再考虑

---

### 四、验证规约引擎(核心中间层 —— 产品的命门)

#### Table stakes
- **F-SPEC-01 规约数据模型**:统一中间表示,与具体执行引擎解耦
- **F-SPEC-02 导航步骤**:用术语库把"去 job 页面"解析成 URL 导航
- **F-SPEC-03 前置步骤**:录制或指引如何让页面进入待验证状态(最难、最该打磨)
- **F-SPEC-04 断言生成**:AI 把"黑底白字""控制台无报错"翻译成具体断言
- **F-SPEC-05 断言类型分类**:功能性(toHaveText/toBeVisible/状态)、显示性(toHaveCSS)、控制台无 error(一等公民)

#### Differentiators
- **F-SPEC-10 断言人工审核入库**:AI 生成断言 → 用户确认 → 才存入用例库(守住准确性)
- **F-SPEC-11 规约可读化展示**:给用户看人话版的"这个用例要验什么",而非纯代码

**Anti-features**
- ❌ 全自动制造任意复杂页面状态 —— 前置靠录制/人工指引,不自动造数

---

### 五、录制能力(参考 ai-ready-pw-codegen)

#### Table stakes
- **F-REC-01 启动录制**:在本地 Chrome 上开始录制用户操作
- **F-REC-02 生成定位器**:录制时用 Playwright codegen 生成稳健定位器(role/text 优先)
- **F-REC-03 录制转规约**:把录制结果转成验证规约(而非裸脚本),成为用例

#### Differentiators
- **F-REC-10 录制时插入断言**:录制过程中提示用户"这里要不要加个断言"
- **F-REC-11 录制片段复用**:前置步骤可录一次,多用例复用

---

### 六、脚本执行(可信主干)

#### Table stakes
- **F-EXEC-01 连接本地 Chrome**:CDP 模式附着到用户已运行的 Chrome
- **F-EXEC-02 执行规约**:把验证规约跑成 Playwright 操作序列
- **F-EXEC-03 断言判定**:确定性执行,pass/fail 不靠模型
- **F-EXEC-04 控制台错误捕获**:监听 console error / page error,作为一条断言
- **F-EXEC-05 截图/快照取证**:每步留证,失败时尤其重要

#### Differentiators
- **F-EXEC-10 失败诊断**:断言失败时,给出"为什么失败"(实际值 vs 期望值 + 截图)
- **F-EXEC-11 重跑/单步调试**:定位用例问题时可单步

---

### 七、AI 视觉 Agent 探索(兜底 —— 仅探索不作裁决)

#### Table stakes(若走此路径)
- **F-AGT-01 Agent 循环**:截图 → 喂视觉模型 → 模型决定下一步 → 执行 → 循环
- **F-AGT-02 探索复现**:尝试在页面上复现问题描述的现象

#### Differentiators
- **F-AGT-10 探索结果人工确认**:Agent 只负责找到现象/收集证据,最终结论人拍板
- **F-AGT-11 探索沉淀**:探索成功后,引导转成脚本规约入库,下次走可信主干

**Anti-features**
- ❌ Agent 作最终 pass/fail 裁决 —— 准确性命脉,不押在模型判断上

---

### 八、验证记录与可追溯

#### Table stakes
- **F-LOG-01 验证记录存储**:每次执行留一条记录(时间、用例、每条断言结果、证据)
- **F-LOG-02 结果展示**:pass/fail 总览 + 详情 + 截图
- **F-LOG-03 历史追溯**:某用例的历史执行曲线

#### Differentiators
- **F-LOG-10 回归对比**:改完再验,对比上次结果

---

### 九、应用外壳与体验

#### Table stakes
- **F-SHELL-01 Electron 应用骨架**:主进程 + 渲染进程(Vue3)+ preload 桥接
- **F-SHELL-02 Chrome 启动辅助**:帮用户用正确参数启动 Chrome(或检测已启动)
- **F-SHELL-03 设置页**:内网模型端点配置、Chrome 路径、数据目录
- **F-SHELL-04 本地数据持久化**:SQLite + 文件系统

## 功能依赖关系

```
F-SHELL(外壳) → F-GL(术语库) ↘
                             F-SPEC(规约引擎) → F-EXEC(执行) → F-LOG(记录)
              F-IN(输入解析) ↗                       ↑
F-REC(录制) ────────────────────────────────────────┘
F-CASE(用例库) ← F-SPEC / F-REC 写入 → F-EXEC 读取
F-AGT(Agent) ← F-EXEC 复用浏览器连接,兜底
```

**建议构建顺序(影响路线图):**
外壳 → Chrome 连接 → 术语库 → 输入解析 → 规约引擎 → 脚本执行 → 录制 → 用例库 → Agent 兜底 → 验证记录深化
