# Project Research — PITFALLS dimension

**Domain:** 前端页面级 AI 验证平台
**Milestone:** Greenfield
**Date:** 2026-07-09

---

## 域专属陷阱(非通用建议)

### P1. Chrome 连接模式选错 —— 产品体验灾难

**陷阱**:用 `launchPersistentContext(userDataDir)` 复用用户登录态。
**为什么是坑**:Chrome 同一 profile 只能有一个实例。若用户日常 Chrome 开着(几乎总是开着),这会失败或打开 `about:blank`(已确认 GitHub issue #35836),要求用户关掉所有 Chrome —— 在内网工作环境里极其打断。
**警告信号**:用户反馈"每次验证都要关浏览器"。
**预防**:从一开始就用 `connectOverCDP`,应用负责引导/代为启动带 `--remote-debugging-port` 的 Chrome。
**Phase 映射**:Phase 1(Chrome 连接)。

---

### P2. 让 LLM 直接生成 Playwright 脚本 —— 脆且不可审核

**陷阱**:AI 把问题单 → 直接吐 Playwright 代码字符串 → 存成 .spec.ts 跑。
**为什么是坑**:
- 脚本不可读(用户看不懂 TS)、不可审核(怎么确认 AI 生成的断言对不对)、不可复用(不同用例间无法组合)
- 模型幻觉会编造不存在的 API/选择器
- 改一个断言要重生成整段
**警告信号**:用户说"我不知道 AI 生成的对不对"。
**预防**:强制走**规约中间层**(结构化 JSON),规约再确定性转译成 Playwright 调用。规约可读、可逐条审核。
**Phase 映射**:Phase(规约引擎)。

---

### P3. AI 断言直接生效、不经人工审核 —— 准确性命脉失守

**陷阱**:AI 生成断言 → 立即入库立即跑 → 用户只看 pass/fail。
**为什么是坑**:模型把"黑底白字"错解成"白底黑字",断言反了,**错的代码反而 pass、对的代码反而 fail**,用户被误导。这是产品最致命的失败模式。
**警告信号**:验证结论与实际相反。
**预防**:AI 断言先进"待审核"态,用户确认才入库。守住准确性这条命脉。
**Phase 映射**:Phase(断言生成)。

---

### P4. 视觉 Agent 作最终裁决 —— 把命脉押给模型

**陷阱**:Agent 探索完直接报"验证通过/失败"。
**为什么是坑**:deepseek-v4-flash 偏快精度存疑;Qwen 即便准,多步视觉判断错误率累积。准确性约束下,模型不能当裁判。
**警告信号**:Agent 给的结论与脚本断言矛盾,或人复看发现 Agent 看错截图。
**预防**:Agent 只输出"证据 + 候选解释",归因结论人定。
**Phase 映射**:Phase(Agent 兜底)。

---

### P5. 前置步骤(让页面进入待验证状态)被低估

**陷阱**:以为有了术语库和定位器就能验,忽略了"怎么让 job 页面出现一个 cancel 状态的 wafer"这种**造数据/走流程**的前置。
**为什么是坑**:这是实际验证中最常卡住的地方。术语库解决"去哪",解决不了"怎么把页面变成那个状态"。很多用例无法自动跑,就卡在前置。
**警告信号**:用例"能导航到页面但验不到",总卡在准备数据。
**预防**:
- 前置步骤优先**录制**(走 Playwright codegen 路线)
- 前置可作为可复用片段,多用例共享
- 不追求全自动造数(v1)
**Phase 映射**:Phase(录制 + 规约前置)。

---

### P6. 内网模型能力假设错误 —— 图像/上下文/稳定性盲点

**陷阱**:默认内网 Qwen/flash 支持图像、上下文够大、调用稳定。
**为什么是坑**:
- 若模型**不支持图像输入**,整个"视觉 Agent 探索"路径直接不可行,需退回纯 DOM/文本
- 若上下文窗口小,塞不下页面 DOM + 多轮对话,Agent 循环会崩
- flash 快但精度差,用在关键判断上不可靠
**警告信号**:Phase 1 调用模型时报 400/不支持 image;或 Agent 探索结论离谱。
**预防**:**Phase 1 第一件事**就是验证模型实际能力(图像?窗口?稳定性?),据此调整架构。脚本主干不依赖图像能力,Agent 兜底才依赖 → 即便图像不可用,主干仍成立。
**Phase 映射**:Phase 1(模型能力探针)。

---

### P7. Playwright 在 Electron 主进程里起不来的坑

**陷阱**:Playwright 在独立脚本里跑得好好的,塞进 Electron 主进程就失败(已确认 StackOverflow #76851093)。
**为什么是坑**:Electron 打包后路径、chromium 二进制查找、asar 内文件访问都会变。Playwright 需要找到浏览器二进制,asar 打包会破坏这个。
**警告信号**:开发模式 OK,打包后 `Browser is not installed` / 路径错。
**预防**:
- chromium 二进制**不打进 asar**,用 `extraResources` 外置
- 用 `connectOverCDP` 连用户的 Chrome(不依赖 Playwright 自带浏览器二进制)—— 这恰好规避了此坑
- 早期就验证打包后能否连接
**Phase 映射**:Phase 1(连接)+ 打包阶段。

---

### P8. CDP 连接的安全/稳定性 —— 多标签页、用户干扰

**陷阱**:应用通过 CDP 接管 Chrome 后,用户在同一 Chrome 里手动操作其他标签页,干扰验证;或验证操作误伤用户当前标签页。
**为什么是坑**:CDP 连的是整个浏览器,不是单个标签页。验证脚本若不限定 target,可能操作到用户正在用的页面。
**警告信号**:验证时用户的别的页面被意外刷新/跳转。
**预防**:
- 验证脚本**显式绑定到特定 tab/page**(按 URL 匹配或新建 tab)
- 验证开始前快照当前标签状态,结束后恢复
- UI 上明确提示"正在控制 Chrome,请勿手动操作被测标签"
**Phase 映射**:Phase(执行控制器)。

---

### P9. 定位器随页面改版失效 —— 用例库腐化

**陷阱**:用 CSS 选择器或 xpath 写死定位器,被测项目一改版,全库失效。
**为什么是坑**:用户改 bug 本就会动页面,定位器一改就坏,用例库迅速变废。
**警告信号**:大量用例突然集体 fail,但功能其实正常。
**预防**:
- 用 Playwright codegen 推荐的**稳健定位器**(role > text > css),避免脆弱选择器
- 定位器与规约解耦,便于批量更新
- 失败诊断时给出"是定位器失效还是真 bug"的区分提示
**Phase 映射**:Phase(录制 + 用例库)。

---

### P10. 把"诊断型"问题当成"断言型"来验 —— 路径错配

**陷阱**:对所有问题单一律走脚本断言。但用户的问题单2(曲线断了,定位前后端)是**诊断型**,脚本 pass/fail 回答不了"问题在哪层"。
**为什么是坑**:用错工具,要么验不出来,要么给出误导性的 pass。
**警告信号**:用户反馈"这类问题单验了等于没验"。
**预防**:输入阶段就**分类路由**(F-IN-04),诊断型走 Agent 探索 + 人工归因,断言型走脚本。
**Phase 映射**:Phase(输入解析 + 分类)。
