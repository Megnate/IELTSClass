## 主要组件边界

### 1. 渲染进程(UI 层)
- **职责**:所有用户交互、表单、列表、可视化
- **边界**:不含任何 Node API、不直接碰数据库、不直接调 Playwright
- **通信**:只通过 preload contextBridge 暴露的白名单 IPC 与主进程通信

### 2. 主进程(核心业务层)
- **职责**:浏览器自动化、LLM 调用、数据持久化、录制控制、规约引擎
- **子模块**:
  - **Playwright 控制器**:维护与本地 Chrome 的 CDP 连接,执行规约
  - **验证规约引擎(Spec Engine)**:把结构化输入转成可执行的步骤+断言序列
  - **LLM 适配层**:封装内网模型调用(经 codex 工具),统一 prompt 模板
  - **录制控制器**:包装 Playwright codegen,把录制流转成规约中间表示
  - **数据层**:better-sqlite3 + 文件证据存储

### 3. 本地 Chrome(外部依赖)
- **关键约束**:必须以 `--remote-debugging-port` 启动,应用通过 CDP 附着
- **不归应用管**:用户的登录态、其他标签页、扩展 —— 应用只读不干扰

### 4. 被测 Web 项目(外部)
- 应用只通过 Chrome 访问它,不直接对接其代码/后端

## 数据流

### 流 A:问题单 → 脚本验证(可信主干)

```
用户粘贴问题单文本
  → [LLM 适配层] AI 解析:提取验证目标/涉及页面/期望结果
  → [术语库] 命中:把"job页面"映射到 URL + 入口
  → [用例库] 检索:是否已有同类用例
     ├─ 命中 → 直接取用例规约
     └─ 未命中 → [LLM] 生成断言 + [录制] 补前置步骤 → 组成新规约
  → [用户审核] 确认断言无误 → 入库
  → [Spec Engine] 规约转 Playwright 步骤序列
  → [Playwright 控制器] 在本地 Chrome 执行 + 断言 + 取证
  → [数据层] 记录验证结果(每条断言 pass/fail + 截图)
  → [UI] 展示结果
```

### 流 B:根因诊断 → Agent 探索(兜底)

```
问题单被判为"诊断型"(如曲线断了,定位前后端)
  → [Agent 循环] 截图 → 视觉模型分析 → 决定操作 → 执行 → 循环
     同时抓:API 响应(网络面板)/ DOM / console / 截图
  → [UI] 汇总证据给用户,人做最终归因
  → (可选)探索成功后引导沉淀成脚本规约
```

### 流 C:需求验收(结构化输入)

```
用户录入/上传需求验收用例(每步已有验收标准)
  → (跳过 AI 推断)直接落成验证规约
  → 后续同流 A 的执行段
```

## 关键架构决策

### A1. CDP 连接模式(非 launchPersistentContext)
- 用 `chromium.connectOverCDP()` 附着到用户**正在运行**的 Chrome
- **理由**:不打断用户工作流,复用登录态,不要求关闭浏览器
- **代价**:应用需引导用户用 `--remote-debugging-port=9222` 启动 Chrome(可由应用代劳)

### A2. 规约引擎作为中间层(核心创新)
- 不让 LLM 直接生成 Playwright 脚本(脆、难审核、难复用)
- 而是 LLM 生成**结构化规约**(JSON),规约再被确定性转译成 Playwright 调用
- **收益**:规约可读、可审核、可存库、可跨用例复用、执行确定性

### A3. 断言审核入库(守准确性)
- AI 生成的断言**不直接生效**,先进"待审核",用户确认才入库
- **理由**:模型可能生成错断言,人工闸门挡住风险

### A4. Agent 只探索不裁决
- 视觉 Agent 的输出是"证据 + 候选解释",pass/fail 归因由人定
- **理由**:模型可能判错,准确性是命脉

### A5. 三进程分层(Electron 标准)
- 渲染进程纯 UI,主进程干重活,preload 做安全桥接
- **理由**:Electron 安全模型;重活在主进程不卡 UI

## 建议构建顺序(依赖驱动)

| 序 | 组件 | 依赖 | 产出可观测价值 |
|----|------|------|----------------|
| 1 | Electron+Vue 外壳 + IPC | 无 | 能跑起应用 |
| 2 | Playwright CDP 连接本地 Chrome | 1 | 能在应用里控制 Chrome |
| 3 | 数据层(SQLite+FS) | 1 | 能存取数据 |
| 4 | 术语库 CRUD + 检索 | 3 | 能维护术语映射 |
| 5 | 输入解析(LLM) | 4 | 能把文本拆成结构化 |
| 6 | 规约引擎 + 脚本执行 | 2,3,5 | **核心闭环跑通** |
| 7 | 用例库(存/检索/执行) | 6 | 可复用 |
| 8 | 录制能力 | 2 | 补前置步骤 |
| 9 | 控制台错误/截图取证 | 6 | 完整证据 |
| 10 | AI 视觉 Agent 兜底 | 2,5 | 探索诊断型 |
| 11 | 验证记录深化/回归对比 | 7 | 长期价值 |

**关键里程碑:序号 6(核心闭环跑通)** —— 这是 MVP 的核心,在此之前都是铺垫。
