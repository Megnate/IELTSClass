讲解：AI随风随风

一、核心认知：决定成败的不是模型本身，而是"Harness"

成功的 Claude Code 大规模部署，决定性因素不仅在于模型本身，更在于围绕模型构建的生态系统。

这个生态系统被作者称为"鞍具"（Harness）——就像一匹好马需要好的鞍具才能发挥潜能一样，Claude Code 也需要精心搭建的鞍具才能在百万行代码的仓库中高效工作。

鞍具组件速查表

组件 是什么 何时加载 最适用场景 常见误区
CLAUDE.md Claude 自动读取的上下文文件 每次会话 项目特定约定、代码库知识 把应放在 Skill 中的可复用专业知识放进来
Hooks 在关键时刻运行的脚本 由事件触发 自动化一致行为、捕获会话学习 用提示词做应该自动运行的事情
Skills 针对特定任务类型的打包指令 按需加载，当相关时 跨会话和项目的可复用专业知识 把所有内容都加载到 CLAUDE.md 中
Plugins 打包的技能、钩子和 MCP 配置 配置后始终可用 在组织内分发有效的设置 让好的设置停留在部落知识中
LSP* 通过语言特定服务器提供实时代码智能 配置后始终可用 符号级导航和类型语言中的自动错误检测 认为它是自动启用的
MCP 服务器 连接外部工具和数据的接口 配置后始终可用 让 Claude 访问无法直接触及的内部工具 在基础配置还没搞好之前就构建 MCP 连接
Subagents* 针对特定任务的独立 Claude 实例 被调用时 将探索与编辑分离、并行工作 在同一会话中运行探索和编辑

二、五层鞍具组件详解

① CLAUDE.md — 最基础、最重要的配置

CLAUDE.md 是每个会话开始时自动读取的文件，相当于给 Claude 的一张"小纸条"。最佳实践是分层配置：

左侧：分层目录结构，每个子目录维护自己的 CLAUDE.md；右侧：Claude 移动时自动叠加上下文的过程。

② Hooks — 自动化的手脚

Hooks 是在关键事件（会话开始/结束、文件写入前/后）触发自动执行的脚本。

会话结束 Hook
自动记录本次会话的教训到经验文件中
文件写入前 Hook
自动运行 lint/format 检查
文件写入后 Hook
自动更新相关文档或生成变更日志
⚠️ 常见误区
用 Hook 写指令指导 Claude（那是 CLAUDE.md 的工作）

③ Skills — 可复用的专业知识包

核心设计理念：渐进式披露（Progressive Disclosure）— 只在需要时才加载特定 Skill，节省上下文空间。

安全审查
需要做安全审查时加载
文档更新
需要更新 API 文档时加载
部署流程
需要执行部署时加载
路径绑定
绑定到 pay-ments/ 目录，仅在此时加载

④ Plugins — 组织级分发的利器

把 Skills、Hooks 和 MCP 配置打包成一个可安装的包，在团队内部一键分发。

⑤ MCP Servers — 连接内部系统的桥梁

让 Claude 能访问内部分析平台、CI/CD 系统、知识库等外部资源。

⚠️ 常见误区：基础配置（CLAUDE.md）没做好就上 MCP。应该 CLAUDE.md → Hooks → Skills → 再 MCP。

构建优先级金字塔

两大额外能力

LSP 集成

让 Claude 进行符号级别的代码导航，而不是基于文本匹配。

  • "转到定义"
  • "查找所有引用"
  • 区分不同语言中的同名函数
  • 在 C/C++ 大型代码库中尤其关键
⚠️ 常见误区：认为 LSP 是自动集成的。需要手动安装对应的插件和语言服务器。

Subagents（子代理）

Subagent 是一个独立的 Claude 实例，拥有自己的上下文窗口。

核心价值：将探索和编辑分离。

  1. 派一个只读子代理"绘制"子系统地图
  2. 主代理基于地图做出精确修改
✅ 避免在同一个上下文中同时做探索和编辑，防止上下文污染

三、核心工作原理：Agentic Search

Claude Code 像一名人类软件工程师那样工作——它遍历文件系统、读取文件、用 grep 搜索、追踪引用。它在本地机器上运行，不需要构建和维护代码索引。

四、三种核心配置模式

模式一

让代码库易于导航

  • CLAUDE.md 精炼分层
  • 从子目录初始化
  • 按子目录限定测试命令
  • 使用 .ignore 排除无关文件
  • 构建 Code Map
  • 运行 LSP
模式二

主动维护配置

为旧模型写的规则可能成为新模型的制约。

建议每 3-6 个月或新模型发布后做一次配置审查。
模式三

分配所有权

  • 基础设-施投入先行
  • 专人负责（Agent Manager）
  • 跨职能工作组

Code Map — 代码库地图

Code Map 是一个轻量级 Markdown 文件，在项目根目录用一行描述一个业务模块（如认证、支付、订单），让 Claude 不用自己摸索「该去哪个模块找代码」。要写的是业务边界清晰的模块，而不是 components/、handlers/ 这类技术子目录。

文章还强调一种分层做法：根目录 CLAUDE.md 提供全局上下文，CODE_MAP.md 列出模块路线图，各子模块下的 CLAUDE.md 负责该模块内的约定与细节——先选模块，再读细节。

配置演进生命周期

旧规则如"每个重构拆分成单文件更改" → 新模型完全可以协调多文件编辑，过时规则应当移除。

六、常见误区 vs 正确做法

三个最重要的决定

1 CLAUDE.md 分层设计
根目录全局 + 子目录本地，按需取用
2 从子目录而非根目录启动
缩小范围，节省上下文
3 基础设施先投
小团队先搭建好体验，再大规模推广

七、案例速览 + 企业落地检查清单

大型零售组织

构建连接内部分析平台的 Skill，打包成 Plugin 分发
启示：标准配置应当共享

企业软件公司

组织范围内部署 C/C++ LSP 集成，作为推广第一步
启示：LSP 在大型代码库中至关重要

Perforce 用户

用 Hook 拦截文件写入执行 p4 edit（后因原生支持移除）
启示：配置需随平台进化更新

支付服务团队

将部署 Skill 绑定到 pay-ments/ 路径
启示：Monorepo 中用路径绑定精准加载

企业落地检查清单

  • 是否有一个小团队负责基础设施搭建？
  • 根目录和子目录是否有分层 CLAUDE.md？
  • 是否配置了基本的 Hooks（lint/format 检查）？
  • 是否有核心任务的 Skills？
  • 验证通过的配置是否打包成 Plugin 分发？
  • 是否有 MCP 连接内部系统？
  • 是否配置了 LSP 集成？
  • 是否有明确的负责人（Agent Manager）？
  • 是否建立了定期审查机制（3-6 月/次）？
